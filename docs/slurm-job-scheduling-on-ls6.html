Chapter 4 SLURM Job Scheduling on LS6

4.1 Overview

On an HPC system, real computational work is normally submitted through a scheduler rather than run directly from the login node. On LS6, that scheduler is SLURM.

This chapter introduces the practical workflow for batch jobs:

  • write a job script
  • request resources
  • submit with sbatch
  • monitor with squeue
  • inspect finished jobs with sacct
  • read output files and diagnose failures

By the end of the chapter, you should be able to submit and interpret your first real SLURM job.

4.2 Mental model: SLURM is a resource manager

Think of SLURM as the system that coordinates fair access to shared compute resources.

You do not simply “grab a node.” Instead, you make a request that describes:

  • what you want to run
  • how many resources you need
  • how long you need them
  • which queue or partition should be used

SLURM then decides when and where your job can run.

4.3 Why clusters use schedulers

Without a scheduler:

  • users would compete chaotically for resources
  • jobs would interfere with each other
  • there would be little accountability for resource use

With a scheduler:

  • jobs are queued
  • resources are allocated intentionally
  • usage is tracked
  • the system remains usable for many people at once

4.4 A basic batch workflow

You write a job script
        ↓
Submit with sbatch
        ↓
Job enters SLURM queue
        ↓
Scheduler allocates compute resources
        ↓
Job runs on compute node(s)
        ↓
Output and job history become available

This is the operational heart of beginner HPC work.

4.5 Your first SLURM script

Create a file:

nano job.slurm

Add:

#!/bin/bash
#SBATCH -J hello_job
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:01:00
#SBATCH -p normal

echo "Hello from compute node"
hostname
date

4.6 Reading the script line by line

4.6.1 Shebang

#!/bin/bash

Tells the system to execute the script using Bash.

4.6.2 Job name

#SBATCH -J hello_job

Gives the job a readable name.

4.6.3 Nodes

#SBATCH -N 1

Requests one node.

4.6.4 Tasks

#SBATCH -n 1

Requests one task.

4.6.5 Wall time

#SBATCH -t 00:01:00

Sets the maximum runtime to one minute.

4.6.6 Partition

#SBATCH -p normal

Requests the normal partition.

Even at this stage, the main idea is becoming visible: a SLURM script is both a command script and a resource request.

4.7 Submit the job

sbatch job.slurm

Typical response:

Submitted batch job 123456

That message does not mean your job has already run. It means the job has been accepted by the scheduler.

4.8 One of the most important beginner corrections

Students often think:

I used sbatch, so my program must already be running.

That is not necessarily true.

sbatch submits the job to the queue. The scheduler then determines when the job can start.

4.9 Monitoring queued and running jobs

Use:

squeue -u $USER

This lets you inspect your current jobs.

Typical fields include:

  • job ID
  • partition
  • job name
  • state
  • elapsed time
  • nodes used

4.10 Common SLURM job states

  • PD = pending
  • R = running
  • CG = completing
  • CD = completed

A pending job is not broken simply because it is pending. It may be waiting for resources.

4.11 Output files as evidence

When a batch job finishes, a file like this often appears:

slurm-123456.out

Inspect it with:

cat slurm-123456.out

or:

cat slurm-*.out

These output files are often the first place to look during debugging.

4.12 What just happened in the first example?

When the simple hello_job runs, you have successfully completed an authentic cluster workflow:

  • wrote a batch script
  • requested resources
  • submitted the request to SLURM
  • allowed the scheduler to place the job
  • executed commands on a compute node
  • collected output from the run

That is the core pattern you will repeat again and again.

4.13 Seeing the distinction between login and compute nodes

Run hostname on the login node and compare it with the hostname printed inside a SLURM job.

That difference makes the cluster architecture concrete.

It helps students see that login access is not the same as compute allocation.

4.14 A longer-running observation job

To observe a job in the queue or running state, edit the script:

#!/bin/bash
#SBATCH -J hello_job
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:02:00
#SBATCH -p normal

echo "Starting job"
hostname
sleep 30
echo "Finishing job"

Submit it again:

sbatch job.slurm

Then check:

squeue -u $USER

This lets you see the difference between job submission and job completion.

4.15 Looking backward with job accounting

Jobs do not stay in squeue forever. Once they finish, you often want history rather than live status.

Use:

sacct

To focus on a specific job:

sacct -j <jobid>

This is valuable for post-run diagnosis.

4.16 Cancelling a job

If a job was submitted incorrectly or no longer needed, use:

scancel <jobid>

Common reasons to cancel a job include:

  • wrong script
  • bad resource request
  • runaway test submission
  • discovering an error immediately after submission

4.17 Common failure patterns

4.17.1 1. The job never starts

Possible reasons include:

  • unrealistic resource request
  • wrong partition
  • excessive wall time
  • waiting for shared resources

4.17.2 2. The job fails immediately

Possible reasons include:

  • typo in a command
  • missing module
  • missing input file
  • bad script logic

4.17.3 3. No output appears where you expected

Possible reasons include:

  • the job is still pending
  • you are looking in the wrong directory
  • output was redirected elsewhere

4.18 Debug from evidence, not guessing

When a job seems wrong, use this sequence.

4.18.1 1. Did it submit?

squeue -u $USER

4.18.2 2. Did it run or finish?

sacct

4.18.3 3. What does the output file say?

cat slurm-*.out

4.18.4 4. Was the environment correct?

Check the job script for module load lines and inspect assumptions.

4.18.5 5. Was the job launched from the correct working directory?

pwd
ls

This checklist prevents you from inventing explanations too early.

4.19 Failure-based learning exercise

Try breaking a script intentionally in one of these ways:

  • misspell a command
  • request an unrealistic combination of resources
  • refer to a file that does not exist
  • load a module that is unavailable

Then diagnose the result using:

  • squeue
  • sacct
  • the output file
  • inspection of the script itself

This exercise is valuable because most HPC competence grows through interpreting evidence after a failure.

4.20 Chapter lab

4.20.1 Part 1: first batch job

  1. Create job.slurm.
  2. Add the minimal example.
  3. Submit with sbatch job.slurm.
  4. Monitor with squeue -u $USER.
  5. Inspect the output file.

4.20.2 Part 2: observe a running job

Add sleep 30, resubmit, and watch the state change.

4.20.3 Part 3: change the resource request

Change from one task to four tasks and compare behavior.

4.20.4 Part 4: load a module inside the job

Add:

module load gcc
which gcc
gcc --version

Submit and inspect the output.

4.21 Best practices to keep now

  • keep scripts small and readable
  • name jobs clearly
  • request realistic time and resources
  • inspect the output after every run
  • save working scripts rather than rebuilding them from memory

4.22 Looking ahead

Once students can submit and monitor jobs confidently, the next natural question is: what exactly are we running on the cluster, and how do we build it? The next chapter moves from job submission to compiling C programs and understanding toolchains.
