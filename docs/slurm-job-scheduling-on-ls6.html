High-Performance Computing on Lonestar6

Chapter 4 SLURM Job Scheduling on LS6

4.1 Overview

On a shared HPC system, real computational work is normally submitted through a scheduler rather than run directly on the login node. On LS6, that scheduler is SLURM.

That sentence is easy to memorize, but the real goal is deeper than memorization. You should understand:

  • why a scheduler exists
  • what a job request really means
  • how a resource request affects queue behavior
  • what evidence to inspect when a job does not behave the way you expected

This chapter develops a practical workflow for beginner and intermediate batch jobs:

  • write a job script
  • request resources deliberately
  • submit with sbatch
  • monitor with squeue
  • inspect finished jobs with sacct
  • interpret output and error files
  • diagnose common failures without guessing

By the end of the chapter, you should be able to submit a real job, explain the meaning of your request, and work through the first layer of job debugging from evidence.

4.2 Mental model: SLURM is a resource manager, not a launch button

A common beginner misunderstanding is to treat sbatch as if it were a special version of bash.

It is not.

When you run a shell script directly, the shell begins executing the script immediately in your current environment.

When you submit with sbatch, you do something different:

  1. you hand SLURM a script
  2. you describe the resources needed for that script
  3. the scheduler decides when those resources are available
  4. the script later runs on one or more compute nodes

So a SLURM job is not just “commands to run.” It is a contract between you and the scheduler:

Here is the work I want to do, here are the resources I believe it needs, and here is how long I think it should take.

That is why a SLURM script is both:

  • a shell script, and
  • a resource request document

This is one of the biggest conceptual shifts in HPC. You are no longer asking only, “What command should I run?” You are also asking, “What is the correct shape of the request for this workload?”

4.3 Why clusters use schedulers

A scheduler exists because HPC systems are shared.

Without a scheduler:

  • users would compete chaotically for access
  • large jobs could crowd out smaller ones
  • there would be no orderly queue
  • usage tracking would be poor
  • fairness would break down quickly

With a scheduler:

  • work is queued
  • resource use is explicit
  • policies can be enforced
  • accounting is possible
  • compute nodes remain organized as a shared service rather than a free-for-all

This is one reason HPC feels different from a personal workstation. On your laptop, you mostly ask, “Can my machine run this?” On a cluster, you also ask, “What is the appropriate request for a shared system?”

4.4 The basic batch workflow

Write job script
      ↓
Submit with sbatch
      ↓
Job enters queue
      ↓
SLURM decides when resources are available
      ↓
Job starts on compute node(s)
      ↓
Output and accounting records become available

This small picture is worth slowing down for. Each step answers a different question.

  • Write job script means you decide what should happen and what resources are needed.
  • Submit with sbatch means you hand that plan to the scheduler.
  • Job enters queue means the job is now waiting to be placed.
  • SLURM decides when resources are available means the system is matching your request to real hardware and current demand.
  • Job starts on compute node(s) means the job finally runs in the environment you asked for.
  • Output and accounting records become available means you can inspect what happened instead of guessing.

This workflow appears again and again in HPC practice. Later topics such as parallel computing, containers, compiled programs, Python workflows, and R workflows still pass through the same scheduler logic.

4.5 Login nodes versus compute nodes

This distinction remains central.

If you forget everything else in the chapter, keep this rule:

Use the login node to prepare work. Use compute nodes to perform work.

A login node is a front door into the cluster. It is meant for light, short, interactive work.

4.5.1 Login nodes are for:

  • editing files
  • reading documentation
  • small test commands
  • moving data
  • compiling small programs
  • submitting jobs

4.5.2 Compute nodes are for:

  • real program execution
  • long-running analyses
  • memory-intensive work
  • parallel workloads
  • training or simulation runs

If a task would monopolize CPU time, consume substantial memory, run for a long time, or scale across multiple cores, it belongs under SLURM.

This matters both technically and socially. Technically, the compute nodes are where the real resources live. Socially, the login nodes are shared by many users, and heavy work there makes the system worse for everyone.

4.6 Your first SLURM script

Create a file:

nano job.slurm

Add:

#!/bin/bash
#SBATCH -J hello_job
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:01:00
#SBATCH -p normal

echo "Hello from compute node"
hostname
date
pwd

This script is intentionally simple. The goal is to make the scheduler behavior visible before the program logic becomes complicated.

At this stage, a simple script is better than a “realistic” but confusing script. You want to see the scheduler, the compute node, and the output file clearly.

4.7 Reading the script line by line

Before rushing to submission, it is worth learning how to read a job script calmly. A good job script is not mysterious. It is a small document that answers two questions:

  1. What resources are being requested?
  2. What commands should run once the job starts?

The #SBATCH lines answer the first question. The ordinary shell commands answer the second.

4.7.1 Shebang

#!/bin/bash

This tells the system to execute the script using Bash.

Even though this line looks familiar from ordinary shell scripting, it still matters here because a SLURM job script is still, at bottom, a shell script.

4.7.2 Job name

#SBATCH -J hello_job

This gives the job a readable name.

Good names matter because later you may have many jobs in the queue or in the accounting records. A name like hello_job is fine for a first test. A name like analysis_run or omp_test_4threads becomes more useful later.

4.7.3 Nodes

#SBATCH -N 1

This requests one node.

At a beginner level, think of a node as one full machine in the cluster. For a simple serial or small shared-memory job, one node is often enough.

4.7.4 Tasks

#SBATCH -n 1

This requests one task.

For a simple serial job, that usually means one process slot. Later, for MPI jobs, this becomes more important because task count often maps to the number of MPI ranks.

4.7.5 Wall time

#SBATCH -t 00:01:00

This tells SLURM the job should need at most one minute.

This is not a casual estimate. It is part of scheduling. If you ask for far more time than needed, the job may wait longer than necessary. If you ask for too little time, the job may be terminated before it finishes.

4.7.6 Partition

#SBATCH -p normal

This requests the normal partition.

A partition is a queue or class of resources. Different systems define partitions differently, but the key mental model is simple:

A partition tells SLURM which pool of resources and policies your job belongs to.

4.8 Submit the job

sbatch job.slurm

Typical response:

Submitted batch job 123456

That message does not mean your program has already run. It means SLURM has accepted the job into the scheduling system.

This is the first beginner correction that really matters:

sbatch submits. It does not promise immediate execution.

A lot of early confusion disappears once you understand this. Submission success and program success are not the same thing.

4.9 Check the queue

Use:

squeue -u $USER

Typical fields include:

  • job ID
  • partition
  • job name
  • state
  • elapsed time
  • number of nodes

If the job starts quickly, you may barely see it. If not, squeue lets you observe the job state.

This command matters because it answers a very practical question:

Is my job still waiting, running, or gone?

That is often the first thing to establish before you do any deeper debugging.

4.10 Common job states

  • PD = pending
  • R = running
  • CG = completing
  • CD = completed

A pending job is not automatically a broken job. It may simply be waiting for resources, waiting behind other jobs, or waiting because your request is harder to place.

If a job is pending, do not jump immediately to the conclusion that the script is wrong. First ask whether the scheduler has simply not placed it yet.

4.11 Output files are evidence

When the batch job finishes, a file such as the following often appears:

slurm-123456.out

Inspect it with:

cat slurm-123456.out

or:

less slurm-123456.out

A strong habit is:

  • do not guess what happened
  • inspect the output file

Your output file is often the fastest path from confusion to evidence.

4.12 What happened in the first example?

When hello_job runs successfully, you have completed a real HPC workflow:

  • wrote a scheduler-aware script
  • requested resources
  • submitted the request
  • waited for SLURM to place the job
  • ran on a compute node
  • captured output in a file

That may seem small, but it is the foundation for almost everything else in the book.

4.13 Resource requests: what you are really asking for

A job request is best understood as a statement of workload shape.

Questions you should learn to ask:

  • Is this a serial job or a parallel job?
  • Does it need one core or many?
  • Does it need one node or several nodes?
  • Is it short or long?
  • Does it require a special software environment?
  • Is the request realistic or inflated?

These questions matter because scheduler performance depends on matching the request to the workload.

4.13.1 Too small a request can cause:

  • slow performance
  • thread or process mismatch
  • jobs that terminate early
  • poor behavior once the code actually scales up

A request that is too small is not necessarily “efficient.” Sometimes it is simply wrong.

4.13.2 Too large a request can cause:

  • unnecessary queue delay
  • wasted allocation
  • poor cluster citizenship
  • confusion when the code uses only a fraction of what was requested

Responsible HPC practice is not “ask for everything.” It is “ask for what the workload actually needs.”

4.14 A better beginner script: make the environment visible

A useful early script is one that prints context:

#!/bin/bash
#SBATCH -J context_job
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:02:00
#SBATCH -p normal

echo "Job ID: $SLURM_JOB_ID"
echo "Job name: $SLURM_JOB_NAME"
echo "Running on host:"
hostname
echo "Working directory:"
pwd
echo "Date:"
date

This helps you see that jobs run in a real environment with real metadata. It also makes debugging easier because you are recording context rather than assuming it.

4.15 The “sleep” demonstration

To observe the queue and running state more clearly, use a job that lasts long enough to inspect.

#!/bin/bash
#SBATCH -J sleep_demo
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:03:00
#SBATCH -p normal

echo "Starting"
hostname
date
sleep 30
echo "Finishing"
date

Submit it:

sbatch sleep_demo.slurm

Then watch:

squeue -u $USER

This is a small but powerful exercise because it makes the scheduler visible.

4.16 Why queue time can change

You may wonder why one job starts immediately while another waits.

The answer is usually some combination of:

  • current system load
  • partition policies
  • how many resources were requested
  • wall time requested
  • job priority
  • the shape of the free resources

This is a good place to build maturity. Queue time is usually not arbitrary. It is the consequence of your request meeting a real shared system.

4.17 Output and error handling

By default, many simple jobs write output to a file such as:

slurm-123456.out

For more deliberate control, you can specify output files directly:

#SBATCH -o job_%j.out
#SBATCH -e job_%j.err

where %j expands to the job ID.

This helps separate standard output from standard error and makes debugging cleaner.

Example:

#!/bin/bash
#SBATCH -J split_output
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:02:00
#SBATCH -p normal
#SBATCH -o split_%j.out
#SBATCH -e split_%j.err

echo "Standard output message"
ls does_not_exist

This is useful because the output and the error now go to different places. When you inspect them, you can tell the difference between:

  • ordinary progress messages
  • failure messages that need attention

4.18 A realistic serial job pattern

Here is a more representative pattern for a serial program:

#!/bin/bash
#SBATCH -J serial_python
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:10:00
#SBATCH -p normal
#SBATCH -o serial_%j.out
#SBATCH -e serial_%j.err

module load python

python analysis.py

This script has all the basic parts of real work:

  • explicit resources
  • explicit environment
  • explicit program launch
  • explicit output handling

The more your scripts look like this, the easier they become to understand and reuse later.

4.19 A realistic compile-and-run pattern

Sometimes the job script is used only to run an already-built executable:

#!/bin/bash
#SBATCH -J c_run
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:05:00
#SBATCH -p normal
#SBATCH -o c_run_%j.out
#SBATCH -e c_run_%j.err

./hello

The separation matters:

  • build small programs on the login node
  • execute real workloads through SLURM

Keeping those roles distinct will make later chapters easier to understand.

4.20 Common beginner mistakes

These mistakes happen often enough that you should expect to see some of them in your own early work. The goal is not to avoid every mistake forever. The goal is to recognize them quickly and diagnose them calmly.

4.20.1 Mistake 1: Assuming sbatch ran the code right away

Correction: sbatch submits to the scheduler. The code may run later.

4.20.2 Mistake 2: Running heavy work directly on the login node

Correction: login nodes are for setup and lightweight tasks.

4.20.3 Mistake 3: Requesting many tasks for a serial program

Correction: more requested resources do not help if the code uses only one process.

4.20.4 Mistake 4: Asking for far too much wall time

Correction: inflated requests can delay scheduling and misrepresent workload needs.

4.20.5 Mistake 5: Ignoring output files

Correction: output and error files are usually the first evidence to inspect.

4.20.6 Mistake 6: Submitting from the wrong directory

Correction: confirm location with pwd and ls before submission.

4.21 A debugging checklist for failed jobs

When a job behaves unexpectedly, work through the following checklist in order. This matters because debugging is much easier when you move from simple evidence to deeper interpretation instead of jumping randomly between ideas.

4.21.1 1. Did the job submit?

Use:

squeue -u $USER

If the job is there, it is either queued or running.

If it is not there, the job may have finished already or failed quickly.

4.21.2 2. What does accounting say?

Use:

sacct

or:

sacct -j 123456

This helps confirm whether the job completed, failed, or was cancelled. It is especially useful when the queue is empty and you are trying to figure out what happened after the fact.

4.21.3 3. What do the output and error files say?

Use:

cat slurm-123456.out

or your custom output and error filenames.

Do not skip this step. The output file often contains the most direct explanation of what went wrong.

4.21.4 4. Was the intended program or file actually present?

Use:

pwd
ls

This checks your working directory assumptions.

A surprising number of job failures are really path failures.

4.21.5 5. Was the required software environment loaded?

If your script assumes Python, GCC, MPI, or another toolchain, verify that the appropriate module commands are present.

The scheduler does not automatically guess the environment you intended.

4.21.6 6. Did the resource request match the program model?

A serial program with four tasks or an OpenMP program without thread settings may behave badly or misleadingly.

If the request and the code do not match, the job may still run, but it may not run in the way you think it does.

4.22 Failure case study 1: misspelled command

Suppose the script contains:

pythno analysis.py

The job may submit successfully, but the output or error file will show that the command is not found.

The key lesson is:

A job can fail even when submission succeeds.

SLURM accepted the job request. The program inside the job then failed.

4.23 Failure case study 2: missing input file

Suppose the script expects input.txt, but the file is not in the working directory:

python analysis.py input.txt

If input.txt is missing, the program may fail immediately.

The fix is usually one of the following:

  • submit from the correct directory
  • use a correct path
  • copy the needed input file first

Again, pwd and ls help.

4.24 Failure case study 3: request does not match workload

Suppose the script requests four tasks:

#SBATCH -n 4

but runs a plain serial script:

python analysis.py

The code itself may still run, but the request does not reflect the workload. This wastes resources and creates confusion.

A good HPC user learns that job design is part of computational discipline.

4.25 Interactive jobs versus batch jobs

Although this chapter emphasizes batch submission, it is still useful to understand the difference conceptually.

Both interactive and batch jobs run on compute resources. The difference is mostly in how you interact with them.

4.25.1 Batch jobs

  • good for repeatable workflows
  • good for production runs
  • good for recorded output
  • good for longer jobs

Batch jobs are the default model for most serious work because they encourage planning, documentation, and repeatability.

4.25.2 Interactive jobs

  • good for exploration
  • good for debugging
  • good for short tests in a compute environment

Interactive jobs are helpful when you need to test behavior live, but they are not a substitute for a well-designed batch workflow.

Learn batch first because it encourages deliberate, documented practice.

4.26 Best practices for SLURM scripts

  • keep scripts readable
  • name jobs clearly
  • request realistic resources
  • separate output and error when useful
  • record environment details for debugging
  • keep useful scripts rather than rewriting from memory each time
  • treat scripts as workflow documents

If you build this habit early, later chapters become easier because you are not inventing your workflow from scratch each time.

4.32 Key takeaways

  • SLURM is a scheduler and resource manager, not a direct execution command
  • a job script is both a shell script and a resource request
  • sbatch submits work to the queue
  • squeue shows queued and running work
  • sacct helps interpret finished jobs
  • output and error files are central debugging evidence
  • thoughtful resource requests are part of responsible HPC practice
