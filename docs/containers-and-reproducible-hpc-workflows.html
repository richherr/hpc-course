Chapter 9 Containers and Reproducible HPC Workflows

9.1 Overview

One of the most common frustrations in computational work is this sentence:

It worked on my machine.

Containers are one of the main ways modern computing tries to solve that problem.

In HPC, containers help package software environments so that workflows become more portable and more reproducible. They are not magic, and they do not remove the need to understand the host system, but they are powerful when used correctly.

This chapter introduces containers as a practical HPC tool. The focus is not on buzzwords. The focus is on how containers fit into real LS6-style workflow.

By the end of the chapter, you should be able to:

  • explain what a container is
  • explain why containers matter in HPC
  • distinguish Docker from Apptainer at a practical level
  • pull or use a simple Apptainer image
  • execute commands inside a container
  • understand why directory binding matters
  • run a containerized command through SLURM
  • recognize common beginner container mistakes

9.2 Mental model: what a container actually is

A container packages:

  • your program
  • required libraries
  • supporting software dependencies
  • environment expectations

This means a container image can act like a portable execution environment.

A simple mental picture is:

Your code
+ supporting software
+ libraries
+ environment assumptions
--------------------------------
Container image

The key idea is not that the container replaces the whole HPC system. It does not. Instead, it packages much of the user-space environment so that software behavior becomes more consistent across contexts.

9.3 Why containers matter in HPC

Containers are useful because they can reduce or manage problems like:

  • missing dependencies
  • conflicting library versions
  • “works on one machine but not another” behavior
  • hard-to-reconstruct software setups
  • brittle custom environments

In research computing, this matters because reproducibility is often not just a convenience issue. It affects whether others can repeat your workflow months later.

9.4 Containers do not eliminate the host system

A very important correction:

A container is not a full replacement for the HPC system around it.

You still depend on:

  • the host kernel
  • the scheduler
  • the filesystem
  • cluster policies
  • the available hardware
  • the way input and output are connected to the environment

So the right mental model is:

  • the host system provides infrastructure
  • the container provides a packaged software environment

Both matter.

9.5 Docker versus Apptainer

9.5.1 Docker

Docker is widely used in development and deployment environments. It is a major tool in software engineering and cloud workflows.

However, on shared HPC systems, Docker often requires privileges or daemon-based behavior that are not a good fit for multi-user clusters.

9.5.2 Apptainer

Apptainer is designed for HPC-style use.

Practical advantages:

  • better suited to shared environments
  • commonly used without root privileges during normal execution
  • integrates naturally into scheduler-based workflows
  • fits better with cluster security models

The practical message is simple:

Docker is part of the broader container world, but Apptainer is the HPC-friendly execution model you are more likely to encounter directly on systems like LS6.

9.6 Checking for Apptainer

First check whether it is already available:

which apptainer

If needed, load it as a module:

module load apptainer

Then verify again:

which apptainer

This mirrors earlier course logic: software availability on HPC systems is often environment-dependent.

9.7 Pulling a simple image

Example:

apptainer pull ubuntu.sif docker://ubuntu

Interpretation:

  • docker://ubuntu identifies the source image
  • ubuntu.sif is the output image file
  • .sif is a common Apptainer image format

This command makes the image a tangible file in your working environment.

9.8 Run a command inside the container

Use:

apptainer exec ubuntu.sif echo "Hello from container"

This is the cleanest way to understand container execution.

You are not “booting a virtual machine.” You are asking Apptainer to run a command inside the packaged environment represented by the image.

9.9 Use an interactive shell

apptainer shell ubuntu.sif

This opens an interactive shell inside the container environment.

Useful exploratory checks include:

which python3
python3 --version
ls /

This helps you see that the container has its own environment logic while still running on the host.

9.10 Binding directories: the most important beginner concept

One of the most common container confusions is visibility of files.

A container image does not automatically expose every host path the way you might expect. Often, you need to bind directories explicitly.

Example:

apptainer exec --bind $PWD ubuntu.sif ls

This makes the current working directory visible inside the container for the command being run.

If your workflow depends on scripts, inputs, or outputs in the current directory, binding is often essential.

9.11 Why binding matters

Without an appropriate bind, a containerized command may not see:

  • your script
  • your input data
  • your working directory
  • your intended output location

That often produces a confusing symptom:

The container starts fine, but it cannot find my files.

Usually the container is not “broken.” The host paths simply were not connected the way you assumed.

9.12 A concrete example with a host script

Suppose script.py exists in the current directory.

You might try:

apptainer exec ubuntu.sif python3 script.py

If the container does not see that host directory, the script may not be found.

A better pattern is:

apptainer exec --bind $PWD ubuntu.sif python3 script.py

This makes the relationship between host files and the container explicit.

9.13 Store images in sensible places

Container images can be large. Good storage decisions matter.

Practical advice:

  • use $WORK or another project-oriented location for sizable images
  • avoid cluttering small quota-limited areas like $HOME with large images
  • think of images as data assets that should be placed deliberately

This connects containers back to the filesystem lessons earlier in the course.

9.14 Running Python inside a container

Example:

apptainer exec ubuntu.sif python3 --version

This is useful because it shows how containers can provide software versions that may not match the default host environment.

9.15 Containerizing a workflow through SLURM

A batch script can run a containerized command just as it runs other programs.

Example:

#!/bin/bash
#SBATCH -J container_test
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:05:00
#SBATCH -p normal
#SBATCH -o container_%j.out
#SBATCH -e container_%j.err

module load apptainer

apptainer exec --bind $PWD ubuntu.sif echo "Running inside container"

Submit with:

sbatch container.slurm

This is an important conceptual bridge:

Containers do not replace SLURM. Containers are often executed under SLURM as part of a scheduled workflow.

9.16 A more realistic containerized script run

Suppose you have a Python script in the current working directory:

#!/bin/bash
#SBATCH -J container_python
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:10:00
#SBATCH -p normal
#SBATCH -o container_py_%j.out
#SBATCH -e container_py_%j.err

module load apptainer

apptainer exec --bind $PWD ubuntu.sif python3 script.py

This is a realistic pattern for working because it combines:

  • scheduler awareness
  • software environment packaging
  • explicit path handling
  • reproducible command execution

9.17 When containers help most

Containers are especially useful when:

  • software dependencies are awkward
  • collaborators need the same environment
  • the host system and project software differ significantly
  • reproducibility matters over time
  • modules alone are not enough or become cumbersome

9.18 When modules may be enough

Not every workflow needs a container.

Sometimes the module system already gives you:

  • a suitable compiler
  • the right Python stack
  • the right MPI stack
  • the needed application version

In those cases, using modules alone may be simpler.

This is an important judgment skill:

Use the tool that fits the workflow. Do not use containers just because they sound modern.

9.19 Common beginner mistakes

9.19.1 Mistake 1: assuming the container sees all host files automatically

Correction: - bind the directories you actually need

9.19.2 Mistake 2: forgetting to load Apptainer

Correction: - verify availability with which apptainer - load the module if needed

9.19.3 Mistake 3: storing large images in the wrong location

Correction: - choose storage locations intentionally

9.19.4 Mistake 4: thinking a container replaces SLURM

Correction: - scheduled execution still goes through SLURM on shared systems

9.19.5 Mistake 5: assuming every needed tool is already inside the image

Correction: - inspect the image and verify commands rather than assuming

9.20 Failure case study 1: file not found inside the container

Symptom: - the script exists on the host - containerized execution says the file is missing

Likely cause: - the working directory was not bound into the container

Fix: - add --bind $PWD or another appropriate bind path

9.21 Failure case study 2: command not found inside the image

Symptom: - python3 or another expected command is missing

Likely cause: - the image does not actually contain that software

Fix: - inspect the image interactively - choose a more appropriate image - build or pull a container that matches the workflow

9.22 Failure case study 3: output appears in the wrong place

Symptom: - the job runs, but expected output is not visible where you expect it

Likely causes: - output path inside the container does not map to the host path - bind paths were incomplete or misunderstood

Fix: - make host/container path relationships explicit - write outputs to bound directories intentionally

9.23 Debugging checklist for container workflows

When a containerized job behaves strangely, ask:

9.23.1 1. Is Apptainer available?

which apptainer

9.23.2 2. Does the image exist?

ls *.sif

9.23.3 3. Can I explore it interactively?

apptainer shell ubuntu.sif

9.23.4 4. Does the command I need actually exist inside the image?

apptainer exec ubuntu.sif which python3

9.23.5 5. Are the needed host directories bound into the container?

Check the --bind logic in the command or script.

9.23.6 6. Does the batch script still make sense as a SLURM job?

Remember that the scheduler and the container are separate layers.

9.24 Best practices

  • load Apptainer explicitly when needed
  • inspect images rather than assuming contents
  • bind host directories intentionally
  • store images in appropriate storage areas
  • keep container commands inside readable SLURM scripts
  • document the image source and runtime assumptions in the project README

9.30 Key takeaways

  • containers package software environments, not the entire HPC system
  • Apptainer is the HPC-friendly container tool in this course context
  • directory binding is one of the most important practical concepts
  • containers often run under SLURM, not instead of SLURM
  • modules and containers solve different workflow problems
  • reproducibility improves when container usage is documented clearly
