High-Performance Computing on Lonestar6

Chapter 2 Introduction to High-Performance Computing on LS6

2.1 Overview

High-performance computing, or HPC, is the practice of using powerful shared computing systems to solve problems that are too large, too slow, or too resource-intensive for an ordinary laptop or desktop computer.

In this book, we use Lonestar6 (LS6) as the working example. The goal is not just to define HPC, but to help you begin using it productively.

By the end of this chapter, you should be able to:

  • describe what a cluster is
  • explain the difference between login nodes and compute nodes
  • identify common LS6 storage locations
  • connect the basic HPC workflow to the commands you will use later

2.2 Why people use HPC

There are three common reasons people move from a local machine to an HPC system.

2.2.1 1. The job is too slow on one machine

A simulation that would take hours or days on a laptop may complete much faster on multiple cores or multiple nodes.

2.2.2 2. The job needs more memory

Some workflows fail locally not because the CPU is too slow, but because the dataset, model, or matrix is too large for the memory available on the machine.

2.2.3 3. The work can be split into parts

Many scientific and data-intensive problems can be broken into pieces and executed in parallel. HPC systems are designed for this style of work.

2.3 Mental model: a cluster is a collection of computers

An HPC system is not one giant computer. It is better to think of it as a coordinated collection of computers working together.

Important terms:

  • node: one computer within the cluster
  • core: one CPU execution unit on a node
  • memory (RAM): memory available to processes or threads running on a node
  • interconnect: the high-speed network that links nodes together

This distinction matters because later chapters will ask you to request resources in terms of nodes, tasks, and cores.

2.4 The most important distinction: login node vs compute node

Beginners often assume that once they log in to the cluster, they are already in the right place to run their real workload. That is not usually true.

2.4.1 Login nodes

Login nodes are used for:

  • editing files
  • navigating directories
  • loading modules
  • compiling small programs
  • moving data
  • submitting jobs

Login nodes are not the place for serious computation.

2.4.2 Compute nodes

Compute nodes are used for:

  • running scheduled jobs
  • interactive compute sessions
  • parallel programs
  • training or simulation workloads
  • large production tasks

2.4.3 Rule of thumb

Do setup on the login node. Do computation on compute nodes.

That single idea prevents many of the worst beginner mistakes.

2.5 A simple cluster workflow

Here is the workflow you will repeat throughout the course:

Laptop
  |
  | ssh
  v
Login node
  |   edit files, inspect directories, load modules
  |   write job scripts
  |
  | sbatch
  v
SLURM scheduler
  |
  v
Compute node(s)
  |
  v
Output files and results

The scheduler in this workflow is SLURM, which will become the center of Chapters 3 and 4.

2.6 Connecting to LS6

A standard SSH connection looks like this:

ssh your_username@ls6.tacc.utexas.edu

After login, you may see a shell prompt like:

[username@login1 ~]$

This prompt tells you several things at once:

  • your username
  • the machine name
  • your current directory

The prompt is not decoration. It gives you context.

2.7 Storage areas you will encounter

Cluster workflows usually involve more than one storage location.

2.7.1 $HOME

Your home directory is a personal, relatively safe place for configuration files, small scripts, and lightweight persistent material.

2.7.2 $WORK

This is often a project-oriented area where you keep larger files, datasets, software builds, and materials that need to persist longer than temporary scratch storage.

2.7.3 $SCRATCH

This is usually a high-performance temporary storage area. It is useful for active computation and temporary job data, but it should not be treated as permanent archival storage.

2.8 Why directories matter so much in HPC

On a laptop, users often rely on clicking around to find files. In an HPC workflow, your current directory strongly influences:

  • where your job looks for input
  • where output files are written
  • whether commands run on the files you intended

That is why pwd and ls show up constantly in beginner workflows.

2.9 Minimal Linux toolkit for starting HPC work

You do not need to master Linux before beginning. You do need a small set of reliable commands.

pwd
ls
cd
mkdir
cp
mv
rm
nano
chmod +x

These commands are enough to support the first real workflow in the course.

2.10 First exercise: create a workspace

mkdir hpc_course
cd hpc_course
pwd
ls

This does four useful things:

  • creates a directory for your course work
  • moves you into that directory
  • verifies where you are
  • confirms what is inside it

A surprising number of job failures come from skipping these checks.

2.11 First shell script

Create a file:

nano hello.sh

Add:

echo "Hello from LS6"

Make it executable:

chmod +x hello.sh

Run it:

./hello.sh

2.12 What just happened?

Even this small example teaches several important HPC habits.

You:

  • created a script file
  • made the file executable
  • ran a lightweight command safely on a login node
  • confirmed that your shell environment is working

This is appropriate login-node work because the task is tiny and brief.

2.13 Common beginner mistakes in Chapter 1

2.13.1 Running heavy work on the login node

This is the most common conceptual mistake. Users often think, “I am logged in, so I can start the analysis.” In HPC, logging in gives you access to the environment, not ownership of shared computational resources.

2.13.2 Losing track of the current directory

If you do not know where you are, you do not really know what your job will act on.

2.13.3 Forgetting execute permission

A file can exist and still fail to run. ls -l and chmod +x are basic troubleshooting tools.

2.13.4 Treating storage areas as interchangeable

Different storage areas serve different purposes. That matters for both performance and data safety.

2.14 Chapter lab

  1. Log in to LS6.
  2. Create a directory named hpc_course.
  3. Move into it and verify with pwd.
  4. Create hello.sh.
  5. Add one echo command.
  6. Make the script executable.
  7. Run it.
  8. Create a second script that prints your username, current directory, and date.

Suggested commands:

whoami
pwd
date

2.15 Reflection questions

  • Why is a login node not the right place for serious computation?
  • What is the difference between logging in and actually running a job?
  • Why does your current directory matter before job submission?
  • Which storage location would you trust for temporary high-volume output, and which would you treat more carefully?

2.16 Looking ahead

In the next chapter, we slow down and build the Linux and environment skills that make later SLURM work easier. The theme of Chapter 2 is simple: good HPC habits begin with filesystem awareness and environment awareness.
