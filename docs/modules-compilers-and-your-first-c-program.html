Chapter 5 Modules, Compilers, and Your First C Program

5.1 Overview

In this chapter, we move from running shell commands and SLURM scripts to building executable software from source code.

This matters because many HPC workflows involve custom code, research software built from source, and toolchains that depend on specific compiler environments.

By the end of this chapter, you should be able to:

  • explain what compilation does
  • load a compiler module
  • verify which compiler is active
  • compile a small C program with gcc
  • run the resulting executable
  • prepare compiled code for SLURM execution

5.2 Mental model: source code becomes an executable

A compiled workflow looks like this:

source code (.c)
      ↓
compiler (gcc)
      ↓
executable binary
      ↓
run on LS6

That simple picture explains a great deal of later HPC practice.

  • source code is human-readable
  • the compiler translates source into machine-executable form
  • the executable is what the system can actually run

5.3 Why compilation matters on clusters

Compilation matters in HPC because:

  • software is not always preinstalled
  • the compiler version can affect behavior
  • libraries and toolchains must often match each other
  • performance can depend strongly on how code is built

For beginners, the main goal is not advanced optimization. It is understanding the workflow clearly.

5.4 Two compilers you will hear about often

  • gcc for C
  • g++ for C++

This chapter focuses on C and gcc, because it gives the cleanest first compiled example.

5.5 Modules first, then compilation

Before compiling, load a compiler environment:

module avail
module load gcc
module list

This reinforces a central HPC idea: software is part of the environment.

5.6 Confirm which compiler you are using

which gcc
gcc --version

This is important because it replaces guessing with evidence.

You now know:

  • whether gcc is available
  • where the executable is coming from
  • which version is active

5.7 Your first C program

Create the file:

nano hello.c

Add:

#include <stdio.h>

int main() {
    printf("Hello from C on LS6\n");
    return 0;
}

This program does very little, but it demonstrates the full compiled workflow.

5.8 Compile the program

gcc hello.c -o hello

Interpretation:

  • hello.c is the source file
  • -o hello names the output executable hello

5.9 Run the executable

./hello

Expected output:

Hello from C on LS6

5.10 Inspect what was created

ls -l

You should see at least:

  • hello.c
  • hello

This is an important moment for beginners: the source file and the executable are not the same thing.

5.11 What just happened?

You:

  • wrote source code
  • invoked a compiler
  • created an executable file
  • ran that executable from the shell

That is the core compiled workflow used across much of scientific computing.

5.12 Optimization flags

A common form is:

gcc hello.c -o hello -O2

Useful levels to know:

  • -O0 = no optimization
  • -O2 = standard practical choice
  • -O3 = more aggressive optimization

For an introductory course, -O2 is a good default example.

5.13 A brief note on C++

A similar pattern exists for C++:

g++ hello.cpp -o hello_cpp

The overall workflow is the same. The compiler changes because the language changes.

5.14 Multiple-file programs

Real projects often have more than one source file.

For example:

main.c
helper.c
helper.h

A simple compile step might be:

gcc main.c helper.c -o program

This introduces an important idea: programs can be built from multiple components.

5.15 Header files

Headers (.h) are used to define interfaces or declarations.

Example:

int add(int a, int b);

Even if students do not yet write large C projects, it helps to see that compilation can involve several files working together.

5.16 Automating builds with make

When builds become repetitive, use a Makefile.

Create:

nano Makefile

Add:

all:
    gcc hello.c -o hello

Then run:

make

The point of this early example is not to teach all of make, but to show how build steps can be automated and standardized.

5.17 Where compilation belongs in the workflow

Good beginner rule:

  • compile on login nodes for small builds
  • run heavier or production computations through SLURM

This keeps the workflow aligned with the earlier distinction between setup and computation.

5.18 Run compiled code through SLURM

Create run.slurm:

#!/bin/bash
#SBATCH -J compile_test
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:05:00
#SBATCH -p normal

./hello

Submit it:

sbatch run.slurm

Now the compiled workflow connects directly back to the scheduler workflow from the previous chapter.

5.19 Common beginner failures

  • forgot module load gcc
  • typo in the source code
  • forgot -o hello
  • tried to run from the wrong directory
  • assumed compilation worked without reading the compiler output

5.20 Debug checklist for compilation work

Use this order:

module list
which gcc
ls -l
gcc hello.c -o hello
./hello

If the compile step fails, read the compiler message slowly. It is often telling you exactly where the problem is.

5.21 Chapter lab

5.21.1 Part 1: compile a C program

  1. Create hello.c.
  2. Compile with gcc hello.c -o hello.
  3. Run ./hello.

5.21.2 Part 2: inspect the files

Use ls -l and note the difference between source and executable.

5.21.3 Part 3: use optimization

Compile with:

gcc hello.c -o hello_opt -O2

5.21.4 Part 4: run through SLURM

Create run.slurm, submit it, and inspect the output file.

5.21.5 Part 5: optional Makefile

Create the minimal Makefile and use make.

5.22 Looking ahead

The next major conceptual jump is from serial execution to parallel execution. Once you understand how to compile and run one program, you are ready to ask how that work can be spread across multiple cores, threads, or nodes using MPI and OpenMP.
