Chapter 5 Modules, Compilers, and Your First C Program

5.1 Overview

In this chapter, we move from running shell commands and submitting SLURM scripts to building executable software from source code.

This matters because much HPC work depends on software that is:

  • written by researchers
  • customized for a lab or project
  • compiled against particular libraries
  • sensitive to the compiler and environment used to build it

That means compilation is not an optional side topic. It is part of the practical workflow of scientific computing.

By the end of this chapter, you should be able to:

  • explain what compilation does
  • distinguish source files from executables
  • load a compiler module
  • verify which compiler is active
  • compile a small C program with gcc
  • explain the difference between compiling and linking at a basic level
  • interpret common build errors
  • prepare compiled code for execution through SLURM

5.2 Mental model: source code becomes an executable

A compiled workflow looks like this:

source code (.c)
      ↓
compiler
      ↓
object code and linking
      ↓
executable binary
      ↓
run on LS6

At a beginner level, this picture is enough to build the right intuition:

  • the source file is written for humans
  • the executable is written for the machine
  • the compiler translates
  • the build process may involve more than one file

One of the most useful early realizations is that editing source code does not automatically change the executable. If you edit the source, you must rebuild the program.

5.3 Why compilation matters on clusters

Compilation matters in HPC because:

  • software is not always preinstalled
  • research code may need to be built from source
  • compiler and library versions must often match
  • toolchains affect performance and reproducibility
  • different module environments can produce different behavior

On a laptop, beginners often get used to one built-in environment. On a cluster, you should think more deliberately:

What compiler am I using? What libraries is it tied to? What environment is active right now?

Those are normal questions in HPC.

5.4 Modules come before compilers

Before compiling, inspect the available software environment:

module avail

Then load a compiler environment:

module load gcc
module list

This reinforces a key course theme:

On an HPC system, software is usually exposed through the environment rather than assumed to be globally fixed forever.

5.5 Verify the compiler instead of guessing

Use:

which gcc
gcc --version

This gives you evidence for:

  • whether gcc is available
  • where the executable is located
  • which version is active

That matters because troubleshooting gets much easier when you know exactly which tool you are using.

5.6 Your first C program

Create a file named hello.c:

nano hello.c

Add:

#include <stdio.h>

int main() {
    printf("Hello from C on LS6\n");
    return 0;
}

This program is intentionally tiny. The point is not complexity. The point is making the compile-and-run pipeline visible.

5.7 Compile the program

gcc hello.c -o hello

Interpretation:

  • hello.c is the source file
  • -o hello names the output executable hello

5.8 Run the executable

./hello

Expected output:

Hello from C on LS6

5.9 Inspect the directory

Use:

ls -l

You should see at least:

  • hello.c
  • hello

This is an important conceptual moment. The source file and executable are different artifacts with different purposes.

5.10 What just happened?

You:

  • wrote human-readable source code
  • invoked a compiler
  • produced machine-executable output
  • ran the resulting program

This is the basic compiled workflow that underlies far more complex HPC software.

5.11 Compile versus link: a useful beginner distinction

Students often use the word “compile” to describe the entire build process. In casual conversation that is common, but it helps to know that there are two ideas present:

5.11.1 Compilation

Turning source code into lower-level object code.

5.11.2 Linking

Combining object code and libraries into a final executable.

At first, these steps are often hidden behind a single command like:

gcc hello.c -o hello

But conceptually, the compiler is doing more than one job on your behalf.

Why does this matter? Because some errors are compile-time errors and others are link-time errors.

5.12 Compile-time errors

These happen when the compiler cannot interpret the source correctly.

Example:

#include <stdio.h>

int main() {
    prinft("Hello\n");
    return 0;
}

Notice the typo: prinft instead of printf.

If you compile:

gcc hello.c -o hello

the compiler will report a problem.

A compile-time error often points to:

  • a typo
  • invalid syntax
  • missing punctuation
  • misuse of the language

5.13 Link-time errors

Link-time errors happen when source files may compile individually, but the final program cannot be assembled correctly.

A common beginner example involves a missing function definition.

Suppose main.c calls a function add, but the file containing add is not included in the build command. Then the compiler may accept main.c, but the linker will complain that the referenced symbol cannot be resolved.

This distinction helps students read build messages more intelligently:

  • syntax problems often mean compilation trouble
  • undefined reference problems often mean linking trouble

5.14 A simple multiple-file example

Imagine the following files:

5.14.1 main.c

#include <stdio.h>
#include "helper.h"

int main() {
    printf("%d\n", add(2, 3));
    return 0;
}

5.14.2 helper.h

int add(int a, int b);

5.14.3 helper.c

int add(int a, int b) {
    return a + b;
}

Build with:

gcc main.c helper.c -o program

Now the interface and implementation are both included.

If you forget helper.c, you may get a link-time error. That is a valuable teaching moment.

5.15 Header files as interfaces

Headers (.h) are commonly used to declare what functions are available. They do not usually hold the full implementation.

At a beginner level, the main takeaway is:

Headers tell the compiler what to expect. Source files provide the actual code.

This makes larger projects more organized and easier to compile piece by piece.

5.16 Optimization flags

A common practical compile command is:

gcc hello.c -o hello -O2

Useful optimization levels to know:

  • -O0 = no optimization
  • -O2 = common practical default
  • -O3 = more aggressive optimization

For an introductory course, -O2 is a reasonable example because it introduces the idea that the build process can affect performance.

The important conceptual point is not to memorize every flag. It is to realize that how a program is compiled can matter.

5.17 Warnings are not noise

Use:

gcc -Wall hello.c -o hello

-Wall asks the compiler to show a wider class of warnings.

Warnings do not always stop the build, but they are often useful clues. Strong beginners learn to read them early instead of ignoring them.

A good principle is:

If the compiler is trying to tell you something, pay attention before the project becomes larger.

5.18 A brief note on C++

A similar pattern exists for C++:

g++ hello.cpp -o hello_cpp

The overall workflow remains familiar:

  • write source
  • invoke compiler
  • produce executable
  • run executable

This chapter uses C because it gives the cleanest first exposure to the build process.

5.19 Makefiles: why build automation exists

As projects grow, repeated build commands become tiresome and error-prone. A Makefile automates the process.

Create:

nano Makefile

Add:

all:
    gcc -Wall -O2 hello.c -o hello

Then run:

make

This is not just a convenience trick. It teaches a valuable lesson:

Build steps are part of the workflow and should be made repeatable.

5.20 A more realistic Makefile example

CC=gcc
CFLAGS=-Wall -O2

all: program

program: main.o helper.o
    $(CC) $(CFLAGS) main.o helper.o -o program

main.o: main.c helper.h
    $(CC) $(CFLAGS) -c main.c

helper.o: helper.c helper.h
    $(CC) $(CFLAGS) -c helper.c

clean:
    rm -f *.o program

Even if students are not yet comfortable writing Makefiles from scratch, reading one helps them understand how real projects are assembled.

5.21 Where to compile

For small builds, compile on the login node.

That is appropriate because compilation is often part of setup work. However, very large builds or special workflows may require different strategies depending on system policy and project needs.

For this course, the safe beginner pattern is:

  • edit on the login node
  • compile on the login node
  • run real jobs on compute nodes via SLURM

5.22 Running compiled code through SLURM

Once the executable exists, a batch script can run it.

Example:

#!/bin/bash
#SBATCH -J c_hello
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:05:00
#SBATCH -p normal
#SBATCH -o c_hello_%j.out
#SBATCH -e c_hello_%j.err

./hello

Submit with:

sbatch run_hello.slurm

This connects Chapter 4 directly to Chapter 3: building and scheduling are different phases of the workflow.

5.23 A practical build-and-run pattern

A typical beginner workflow might look like this:

module load gcc
gcc -Wall -O2 hello.c -o hello
ls -l hello
./hello
sbatch run_hello.slurm

This order is instructional gold because it separates concerns clearly:

  1. load environment
  2. build executable
  3. test executable lightly
  4. run real work through the scheduler

5.24 Common compiler and build errors

5.24.1 Error type 1: syntax problem

Symptoms: - missing semicolon - typo in a function name - unbalanced braces or parentheses

Response: - read the line number - inspect nearby lines - fix syntax before trying other theories

5.24.2 Error type 2: missing header

Symptoms: - compiler says it cannot find a header file

Response: - check the include line - confirm the file exists - confirm search paths if using nonstandard locations

5.24.3 Error type 3: undefined reference

Symptoms: - the compiler gets farther, then the linker complains

Response: - check whether all needed .c files were included - check whether required libraries were linked

5.24.4 Error type 4: command not found

Symptoms: - shell cannot find gcc

Response: - check module load gcc - run which gcc

5.25 Failure case study 1: typo in source code

Suppose you accidentally write:

#include <stdio.h>

int main() {
    printf("Hello\n")
    return 0;
}

The missing semicolon after printf will prevent compilation.

Lesson: - the executable cannot update until the source is valid

5.26 Failure case study 2: typo in the compile command

Suppose you run:

gcc hell.c -o hello

but the file is actually named hello.c.

Lesson: - sometimes the mistake is not in the source, but in the shell command itself

Use:

ls

to confirm filenames rather than relying on memory.

5.27 Failure case study 3: object exists, executable is stale

Suppose you edit hello.c, forget to rebuild, and run ./hello.

You may see old behavior and think the edits failed.

What really happened is simpler:

  • source changed
  • executable did not

Lesson: - rebuilding is part of the development loop

5.28 Reading compiler messages calmly

Compiler output can look intimidating because it is dense. A good beginner habit is to read it in layers:

  1. identify the first real error
  2. find the file and line number
  3. inspect the local code near that line
  4. fix one issue at a time
  5. recompile

Do not scroll immediately to the bottom and panic. Often the first meaningful error is the most useful one.

5.29 Good habits for build workflows

  • load the compiler module explicitly
  • check which gcc and gcc --version
  • keep source files organized
  • rebuild after edits
  • use -Wall
  • automate repetitive builds with make
  • keep run scripts separate from source files
  • treat build commands as part of reproducible workflow documentation

  1. Create hello.c
  2. Load gcc
  3. Compile with gcc hello.c -o hello
  4. Run ./hello

5.34 Stretch goal

Create a small program that reads input, performs a calculation, and writes output. Then:

  • compile it with warnings enabled
  • create a SLURM script to run it
  • store source in src/
  • store the job script in jobs/
  • document the build and run commands in a README.md

5.35 Key takeaways

  • source code and executables are different artifacts
  • compiler modules are part of the HPC environment
  • compile-time and link-time errors are different kinds of failures
  • warnings are useful, not just noise
  • Makefiles turn repeated build steps into reproducible workflow
  • small programs can be built on the login node, but real execution belongs under SLURM
