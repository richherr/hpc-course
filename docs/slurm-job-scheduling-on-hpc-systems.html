Chapter 4 SLURM: Job Scheduling on HPC Systems

4.1 Overview

On an HPC system, you do not usually run serious computational work directly on the command line.

Instead, you submit jobs to a scheduler.

On Lonestar6, that scheduler is SLURM.

This chapter introduces:

  • what a scheduler does
  • how to write a SLURM script
  • how to submit and monitor jobs
  • how to debug common problems

4.2 Mental Model: Renting Compute Power

Think of SLURM as a resource manager.

You request:

  • nodes
  • tasks
  • time
  • sometimes memory or other constraints

Then the system decides when and where your job runs.

4.3 The Basic Workflow

User → Job Script → SLURM → Compute Node → Output File

4.4 Why a Scheduler Exists

Without a scheduler:

  • users would compete directly for resources
  • jobs would collide
  • the cluster would become disorganized

With a scheduler:

  • resources are allocated fairly
  • jobs are queued
  • usage is tracked

4.5 What Is a Job?

A job is a request that describes:

  • what program to run
  • how many resources it needs
  • how long it may run
  • what environment it needs

4.6 Your First Job Script

Create a file:

nano job.slurm

Add this content:

#!/bin/bash
#SBATCH -J hello_job
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:01:00
#SBATCH -p normal

echo "Hello from compute node"
hostname
date

4.7 Breaking Down the Script

4.7.1 Shebang

#!/bin/bash

Tells the system to use Bash.

4.7.2 Job name

#SBATCH -J hello_job

This gives your job a readable name.

4.7.3 Number of nodes

#SBATCH -N 1

Request one node.

4.7.4 Number of tasks

#SBATCH -n 1

Request one task.

4.7.5 Wall time

#SBATCH -t 00:01:00

Maximum runtime of one minute.

4.7.6 Partition

#SBATCH -p normal

Request the normal queue or partition.

4.8 Submitting a Job

sbatch job.slurm

You should see a message similar to:

Submitted batch job 123456

4.9 Monitoring Jobs

4.9.1 Show your queued and running jobs

squeue -u $USER

This shows:

  • job ID
  • partition
  • job name
  • state
  • elapsed time
  • nodes used

4.10 Common Job States

  • PD → pending
  • R → running
  • CG → completing
  • CD → completed

4.11 Output Files

When the job finishes, you will usually see:

slurm-123456.out

View it with:

cat slurm-123456.out

Or more generally:

cat slurm-*.out

4.12 Mental Model: What Just Happened?

You:

  • wrote a batch script
  • submitted it with sbatch
  • SLURM placed it in the queue
  • the scheduler assigned resources
  • the script ran on a compute node
  • output was written to a file

4.13 Important Distinction: Login Node vs Compute Node

When you type hostname on the login node, you will see a login-node name.

When the same command runs inside a SLURM job, it shows the compute node that actually executed your work.

4.14 Adding a Delay to Observe the Queue

Edit your script:

#!/bin/bash
#SBATCH -J hello_job
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:02:00
#SBATCH -p normal

echo "Starting job"
hostname
sleep 30
echo "Finishing job"

Submit it:

sbatch job.slurm

Then monitor:

squeue -u $USER

4.15 Common Mistakes

4.15.1 1. Job never starts

Possible reasons:

  • requested too many resources
  • requested the wrong partition
  • requested too much wall time

4.15.2 2. Job fails immediately

Possible reasons:

  • script syntax error
  • bad command
  • missing module
  • missing input file

4.15.3 3. No output appears

Possible reasons:

  • job is still queued
  • wrong working directory
  • output redirected elsewhere

⚠️ Very Common Beginner Error

Students often assume sbatch runs the job immediately.

It does not.

It submits the job to the scheduler.

4.16 Inspecting Completed Jobs

A useful accounting command is:

sacct

A more targeted version is:

sacct -j <jobid>

4.17 Cancelling a Job

If you need to stop a job:

scancel <jobid>

4.18 Minimal Serial Job Example

#!/bin/bash
#SBATCH -J serial_test
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:05:00
#SBATCH -p normal

echo "Running serial test"
python myscript.py

4.19 Example with Module Loading

#!/bin/bash
#SBATCH -J gcc_test
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:05:00
#SBATCH -p normal

module load gcc

echo "Compiler path:"
which gcc
gcc --version

4.20 Example with Multiple Tasks

#!/bin/bash
#SBATCH -J multitask_test
#SBATCH -N 1
#SBATCH -n 4
#SBATCH -t 00:05:00
#SBATCH -p normal

echo "Testing multiple tasks"
hostname

4.21 Mental Model: Resources

A good beginner mental model is:

A SLURM script is a contract:

“Here is what I want to run, and here are the resources I need.”

4.22 Lab 3 — First SLURM Jobs

4.22.1 Part 1: Your first job

  1. Create job.slurm
  2. Add the minimal script
  3. Submit with:
sbatch job.slurm
  1. Monitor with:
squeue -u $USER
  1. View output file

4.22.2 Part 2: Observe a running job

Modify the script to include:

sleep 30

Resubmit and monitor the job while it runs.

4.22.3 Part 3: Change resource request

Change:

#SBATCH -n 1

to:

#SBATCH -n 4

Resubmit and compare behavior.

4.22.4 Part 4: Load a module in the job

Modify the script to include:

module load gcc
which gcc
gcc --version

Submit and inspect output.

4.23 Debugging Checklist

When a job fails, ask:

4.23.1 1. Did it submit?

squeue -u $USER

4.23.2 2. Did it complete or fail?

sacct

4.23.3 3. What does the output file say?

cat slurm-*.out

4.23.4 4. Did I load the right modules?

Check the script itself and your environment assumptions.

4.23.5 5. Am I in the correct directory?

pwd
ls

4.24 Failure-Based Exercise

Try intentionally breaking a script in one of these ways:

  • misspell a command
  • load a module that is unavailable
  • request an unrealistic resource combination
  • refer to a missing file

Then use:

  • sacct
  • output files
  • script inspection

to diagnose what happened.

4.25 Best Practices

  • keep scripts small and readable
  • request realistic wall time
  • name jobs clearly
  • inspect output every time
  • use version-controlled job scripts when possible

4.26 Conceptual Bridge

Up to this point, students have learned:

  • how to access the system
  • how to work in Linux
  • how to manage their environment
  • how to submit jobs

The next step is to understand how software is actually built and prepared for these environments.

4.27 Looking Ahead

In the next chapter, we will move into compiling software:

  • using gcc and g++
  • understanding build steps
  • working with modules and toolchains
  • preparing programs for HPC execution

4.28 Key Takeaways

  • Serious computation is submitted through SLURM
  • A job script describes both commands and resources
  • sbatch submits work
  • squeue monitors queued and running jobs
  • sacct helps inspect job history
  • Output files are central to debugging
