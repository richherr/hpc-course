Chapter 3 Linux Fundamentals and Environment Awareness

3.1 Overview

This chapter builds the Linux foundation needed for real HPC work. Many job failures that appear to be “cluster problems” are actually basic environment problems:

  • wrong directory
  • missing file
  • missing execute permission
  • wrong module environment
  • command not found

By the end of this chapter, you should be able to:

  • interpret the output of ls -l
  • change file permissions for common cases
  • inspect and reason about environment variables
  • understand what PATH does
  • load, list, and unload modules
  • use a practical debugging checklist before submitting work

3.2 Mental model: an HPC system is a shared environment

On a personal laptop, users often assume that their preferred software, file layout, and editor are just “there.” On an HPC system, the environment is more explicit.

You are working in a space that is:

  • multi-user
  • resource-managed
  • environment-driven
  • structured for repeatability

That means your workflow should be deliberate. You should know:

  • where you are
  • what files are present
  • what software is loaded
  • what commands are available in your environment

3.3 Inspecting the filesystem

The first two commands to trust are:

pwd
ls -l

3.3.1 pwd

Prints the current working directory.

3.3.2 ls -l

Lists files with useful details such as permissions, owner, group, size, and modification time.

An example permission string looks like this:

-rwxr-xr--

A good beginner interpretation is:

  • the file exists
  • it has an owner, group, and permission settings
  • some users may execute it, some may not

3.4 File permissions in practice

Permissions matter constantly in Linux-based HPC workflows.

You do not need a full theory of Unix security to start working productively. You do need the operational basics.

3.4.1 Common actions

Make a file executable:

chmod +x script.sh

Remove write permission:

chmod -w file.txt

Use numeric mode:

chmod 755 script.sh

A helpful mental model for 755 is:

  • owner: read, write, execute
  • group: read, execute
  • others: read, execute

3.5 Why permissions matter on clusters

Common failures include:

  • a script exists but will not execute
  • a file cannot be edited because it is not writable
  • a shared file has the wrong permissions for collaborators

When a script “should work” but does not, ls -l is one of the fastest checks you can make.

3.6 Environment variables

Environment variables define part of the context in which commands run.

View them with:

env

Two important examples are:

echo $HOME
echo $PATH

3.6.1 $HOME

Usually points to your home directory.

3.6.2 $PATH

Tells the shell where to search for executable programs.

If a program is not on your PATH, typing its name may produce:

command not found

3.7 A practical mental model for PATH

Think of PATH as a list of directories the shell searches when you enter a command name.

If gcc is on your PATH, you can type:

gcc --version

If not, the shell may not know where to find it.

A temporary way to add your own directory is:

export PATH=$HOME/bin:$PATH

This prepends $HOME/bin to the front of the search path for the current shell session.

3.8 Modules: software is part of the environment

Clusters often use a module system rather than making every software package globally visible at all times.

That means software is often loaded into the environment when needed.

Common commands:

module avail
module load gcc
module list
module unload gcc

3.9 Why modules matter

On an HPC system, different projects may need:

  • different compiler versions
  • different MPI libraries
  • different Python stacks
  • different container runtimes

Modules help manage this complexity.

Instead of assuming “the system version” is correct, users load an appropriate environment for the job.

3.10 Example: loading a compiler

module load gcc
which gcc
gcc --version

This sequence is valuable because it does three things:

  • loads the compiler module
  • shows which executable will run
  • shows the version now active in the shell

3.11 Practical file inspection tools

A few file-related commands are worth using early.

Find a file:

find . -name "example.txt"

View a file:

cat example.txt
less example.txt
head example.txt
tail example.txt

These tools are especially helpful when checking logs, outputs, and small configuration files.

3.12 Piping commands together

One of the most useful Linux habits is combining small commands.

cat file.txt | grep "error"

This is a classic pipeline: one command produces output, the next filters it.

You do not need to become an expert in shell pipelines immediately, but you should recognize that Linux tools are designed to be chained together.

3.13 Disk awareness matters

HPC users need at least basic awareness of storage usage.

Check filesystem space:

df -h

Check directory sizes:

du -sh *

These commands help prevent confusion when jobs fail because of full storage areas or because output is larger than expected.

3.14 Process awareness

Sometimes you need to inspect or stop your own processes.

List your processes:

ps -u $USER

Kill a process:

kill <pid>

This becomes more useful later when dealing with editors, accidental runs, or interactive sessions.

3.15 Optional connection to R

For students who use R heavily, it is useful to remember that environment information can also be inspected from inside R.

Sys.getenv("PATH")

And shell commands can be invoked from R when appropriate:

system("echo $HOME")

That said, it is still important to understand the shell environment directly rather than relying only on IDE tools.

3.16 A debugging mindset for Linux-based HPC work

When something goes wrong, start with evidence.

Ask these five questions in order:

3.16.1 1. Where am I?

pwd

3.16.2 2. What files are here?

ls -l

3.16.3 3. Do I have the right permissions?

ls -l script.sh

3.16.4 4. Is the program actually available?

which gcc

3.16.5 5. Is my environment what I think it is?

module list

This checklist will reappear in later chapters because it prevents premature guessing.

3.17 Common beginner mistakes in Chapter 2

  • forgetting chmod +x
  • assuming PATH is correct without checking
  • forgetting to load a needed module
  • filling up $HOME and wondering why workflows behave strangely
  • treating missing commands as mysterious system failures instead of environment issues

3.19 Stretch goal

Create a custom bin/ directory in your home space, add it to PATH, place a small script in that directory, and run it without typing the full path.

3.20 Looking ahead

The next chapter turns these Linux and environment skills into a real HPC workflow. Once you can reliably manage directories, permissions, and modules, you are ready to write a SLURM script and submit work to the scheduler.
