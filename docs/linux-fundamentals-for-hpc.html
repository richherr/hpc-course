Chapter 3 Linux Fundamentals for HPC

3.1 Overview

In this chapter, you will build the Linux habits needed to work effectively on a shared HPC system.

Many beginners think Linux is just a collection of commands to memorize. That is not the right way to approach it. On an HPC system, Linux is the environment in which you:

  • find and organize your files
  • inspect permissions
  • prepare software environments
  • launch and monitor work
  • diagnose failure when something goes wrong

So the real goal of this chapter is not just command memorization. The real goal is to help you become environment-aware.

You should finish this chapter being able to answer questions like:

  • Where am I?
  • What files are here?
  • Do I have permission to use this file?
  • Is the command I need actually available?
  • Is my software environment loaded correctly?
  • Where did my output go?
  • Am I using the right storage location?

These questions come up constantly in real HPC work.

3.2 Why Linux matters in HPC

On a laptop, you can often get away with clicking around until you find what you need. On an HPC system, that habit breaks down quickly.

You are working in:

  • a remote environment
  • a shared filesystem
  • a shell-based workflow
  • a scheduler-managed system
  • a software environment controlled by modules

That means Linux skills are not optional background knowledge. They are part of the core workflow.

If you lose track of your current directory, misunderstand permissions, or assume a command exists when it does not, your job may fail before the “real” scientific work even begins.

This is why Linux fundamentals belong early in the book. Many later errors are really Linux errors in disguise.

3.3 Mental model: you are working in a shared environment

Unlike a personal laptop, an HPC system is:

  • multi-user
  • policy-driven
  • environment-sensitive
  • designed for repeatable workflows

That changes what “good habits” mean.

On a personal machine, a messy directory or vague file naming may be annoying but manageable.

On a shared cluster, messy habits lead to:

  • lost files
  • wrong paths
  • broken scripts
  • quota problems
  • confusing job output
  • workflows that are hard to reproduce later

A strong habit on HPC is:

inspect first, act second

That one habit will prevent many avoidable mistakes.

3.4 The Linux filesystem

A simplified picture looks like this:

/
├── home/
│   └── your_username/
├── work/
│   └── project_space/
├── scratch/
│   └── temporary_files/

You do not need to memorize every system directory. You do need a clear working understanding of where your own files live and what each major storage area is for.

3.5 A practical storage model

A useful beginner model is:

  • /home or $HOME → small, stable, personal files
  • /work or $WORK → active project files and larger working data
  • /scratch or $SCRATCH → high-performance temporary data

These names can vary somewhat by system, but the underlying ideas are common.

3.5.1 $HOME

Use it for:

  • shell configuration
  • small scripts
  • notes
  • small code files
  • personal settings

Do not treat it as a dumping ground for large datasets, container images, or long-term accumulation of bulky outputs.

3.5.2 $WORK

Use it for:

  • active project directories
  • larger inputs
  • files you expect to revisit regularly
  • project-oriented working material

3.5.3 $SCRATCH

Use it for:

  • temporary high-performance workflow data
  • files created during active jobs
  • large intermediate outputs that do not need permanent storage

Scratch is useful because it is often better suited for active job I/O, but you should never forget that it is temporary by design.

3.6 Where am I?

The first Linux question you should learn to ask is:

pwd

This prints your present working directory.

This command is deceptively important. Many later failures are really just “I was in the wrong directory.”

If you ever think:

  • “Why can’t it find my file?”
  • “Why did the output show up there?”
  • “Why is my script failing to open the input?”

start with pwd.

3.7 What is here?

Use:

ls

For more detail:

ls -l

For hidden files too:

ls -la

A strong habit is to use ls -la more often than plain ls, because hidden files and permissions often matter in real workflows.

3.8 Reading ls -l

Example output may look like:

-rw-r--r-- 1 user group size date filename

That one line tells you several useful things:

  • file type
  • permissions
  • owner
  • group
  • size
  • timestamp
  • filename

Instead of treating ls -l as clutter, learn to read it as diagnostic information.

3.9 File permissions

Every file and directory has permissions for:

  • owner
  • group
  • others

Each of those can have:

  • read (r)
  • write (w)
  • execute (x)

3.9.1 Visual reminder

-rwxr-xr--
│││││││││
│││││││└── others
│││││└──── group
││└──────── owner

3.10 Why permissions matter

Permissions directly affect whether you can:

  • run a script
  • edit a file
  • enter a directory
  • read an input file
  • share work safely

In HPC, permissions matter because your jobs do not magically ignore filesystem rules. If the script cannot read a file, the scheduler will not fix that for you.

3.11 Common permission operations

3.11.1 Make a script executable

chmod +x script.sh

Use this when a file should behave like a runnable script rather than ordinary text. On Linux, the system does not assume that every text file containing commands is meant to be executed. The execute bit makes that intention explicit.

This is one of the most common beginner fixes on HPC systems, especially when a job script or helper script exists, looks correct, but refuses to run.

3.11.2 Remove write permission

chmod -w file.txt

This is useful when you want to protect a file from accidental edits. For example, once an input file or reference script is finalized, removing write permission can reduce the chance that you accidentally change it during a later session.

3.11.3 Use numeric mode

chmod 755 script.sh

Numeric permission notation is compact and common in documentation, scripts, and examples. You do not need to love it immediately, but you do need to recognize what it is doing when you see it in real workflows.

That means:

  • user: rwx
  • group: r-x
  • others: r-x

3.12 A real permission example

Suppose you create a shell script called run_analysis.sh and then type:

./run_analysis.sh

If the system complains, do not immediately rewrite the script.

Check:

ls -l run_analysis.sh

If the execute bit is missing, the fix is:

chmod +x run_analysis.sh

This is a good example of Linux debugging on HPC:

  • do not guess
  • inspect the file
  • make the smallest needed correction
  • rerun

3.13 Directories also have permissions

Beginners often focus only on file permissions, but directories matter too.

If you cannot enter a directory, list its contents, or write files into it, directory permissions may be the reason.

So when something feels “missing,” do not think only about the file. Think about the directory path that leads to it.

3.14 Moving around the filesystem

3.14.1 Change directory

cd my_directory

This moves your shell into another directory. Because so many commands act on the current working directory, cd quietly changes the meaning of everything you do next.

3.14.2 Go up one level

cd ..

This moves you to the parent directory. It is especially useful when you realize you have gone too deep into a project tree and want to step back up without typing a full path.

3.14.3 Go to your home directory

cd ~

This is a reliable way to reset yourself to a familiar location. When you feel disoriented, going home and checking pwd is often better than guessing where you are.

3.14.4 Return to the previous directory

cd -

This last one is especially useful when you are switching between two working locations.

3.15 Creating and organizing directories

3.15.1 Make a directory

mkdir project1

This creates a new directory with the name you provide. On HPC systems, deliberate directory structure matters because it helps separate scripts, data, logs, and outputs before your workflow grows messy.

3.15.2 Make nested directories

mkdir -p project1/data project1/output project1/scripts

The -p option is helpful because it creates intermediate directories when needed and does not complain if some of them already exist. This makes it a good pattern for setting up a project structure quickly and repeatably.

This is a good early habit because project organization matters. Strong directory structure makes debugging easier later.

A clean small project structure may look like:

project1/
├── data/
├── output/
├── scripts/
└── notes/

Even small projects benefit from this kind of organization.

3.16 Copying, moving, and deleting

3.16.1 Copy a file

cp file1.txt file2.txt

This makes a duplicate of a file. It is useful when you want to preserve an original version before experimenting, editing, or moving the file elsewhere.

3.16.2 Copy a file into a directory

cp file1.txt project1/data/

This keeps the original file and places a copy into the target directory. That is a good pattern when you are staging input files for a project without destroying the original location.

3.16.3 Move or rename a file

mv oldname.txt newname.txt

mv is used both for renaming and for moving files. That dual role is useful, but it also means you should pay attention to the destination because the same command can either tidy your naming or relocate the file entirely.

3.16.4 Remove a file

rm file1.txt

This permanently removes the file in the usual command-line sense. Because there is often no easy undo, it is worth checking the filename carefully before pressing Enter.

3.16.5 Remove a directory and its contents

rm -r old_directory

The -r option means recursive removal. That is powerful and potentially dangerous because it removes the directory and everything inside it, not just the top-level folder name.

Be careful with rm. On many Linux systems, deleting a file this way does not send it to a recycle bin.

That means inspection matters before deletion.

3.17 Viewing file contents

3.17.1 Print a short file

cat file.txt

Use cat when the file is short enough to read in one burst. It is convenient for quick checks, but not ideal for long output because it dumps everything to the screen immediately.

3.17.2 Read interactively

less file.txt

Use less when you want to scroll, search, and inspect a file without flooding the terminal. It is especially useful for job output, log files, and other text that may be longer than one screen.

3.17.3 Inspect the beginning

head file.txt

head is useful when you want a quick look at the first lines of a file. That often helps you confirm whether the file is the one you expected, whether it has a header, or whether the beginning of a log already shows a problem.

3.17.4 Inspect the end

tail file.txt

tail is useful because many programs report final status messages, recent progress, or fatal errors near the end of the file. It is often the fastest way to inspect the newest information in a log.

These commands become extremely useful once you start reading job output files.

A strong habit is:

  • use head when you want the beginning quickly
  • use tail when you want the last lines of a long output file
  • use less when you want to scroll and search interactively

3.18 Searching for files

Use:

find . -name "file.txt"

This command is one of the best ways to recover from a “Where did that file go?” problem.

If output seems to be missing, find is often more useful than randomly changing directories and listing files.

3.19 Searching inside files

Use:

grep "error" file.txt

This searches for lines containing a pattern.

Example:

grep "warning" sample.log

This matters in HPC because output and error files can become long. You do not always want to read every line from top to bottom before looking for useful clues.

3.20 Pipes: connecting commands

A pipe sends the output of one command into another command.

Example:

cat file.txt | grep "error"

A better mental model is:

one command produces information, the next command filters it

This is valuable because Linux tools often work best in combination rather than isolation.

You will use this pattern often for output inspection.

3.21 Environment variables

Environment variables hold information about the current shell environment.

Examples:

echo $HOME
echo $PATH
echo $SHELL

These variables are not just trivia. They tell you where your home directory is, how commands are discovered, and what shell is interpreting your session. When something behaves differently from what you expected, environment variables are often part of the explanation.

This information helps define:

  • where executables are found
  • where your home directory is
  • what shell you are using
  • how your environment behaves

3.22 Why environment variables matter

On HPC systems, many tools are activated through the environment rather than installed in one universal location.

That means command behavior may depend on:

  • loaded modules
  • shell startup files
  • path settings
  • temporary environment choices

So if something works in one shell but not another, the environment may be the reason.

3.23 Understanding PATH

PATH tells Linux where to search for commands.

Check it with:

echo $PATH

If a program is not in one of those directories, the shell may answer:

command not found

That does not always mean the program is absent from the system. It may mean the environment is not configured to find it.

3.24 Add something to PATH temporarily

export PATH=$HOME/bin:$PATH

This adds your personal bin directory to the front of the search path for the current shell session.

That is useful when you write your own small scripts and want to run them by name.

3.25 Diagnosing a missing command

Suppose you try:

gcc --version

and see:

bash: gcc: command not found

Do not guess wildly.

Instead, work through a reasoning chain:

  1. Is the relevant module loaded?
  2. What does module list show?
  3. What does which gcc show?
  4. What is in PATH?

This is much stronger than changing several unrelated settings at once.

3.26 Modules: software environments on HPC systems

HPC systems often avoid putting all software into one global environment.

Instead, they use modules.

This matters because different projects may need:

  • different compiler versions
  • different MPI implementations
  • different library stacks
  • different dependency combinations

Modules let you switch environments deliberately.

3.27 Basic module commands

3.27.1 Show available modules

module avail

This helps you answer the question, “What software environments could I activate right now?” It is often the first command to run when you know what tool you need but do not yet know its exact module name.

3.27.2 Load a module

module load gcc

This changes your current environment so that the requested software stack becomes active. Loading a module is not just “opening a program.” It often updates PATH, libraries, and related environment settings at the same time.

3.27.3 Show loaded modules

module list

This tells you what software environment is currently active. It is one of the best quick checks to run when a command behaves differently than you expected or when you need to document the environment you used.

3.27.4 Unload a module

module unload gcc

This removes the module from the current environment. It is useful when you want to switch toolchains, reduce environment clutter, or verify that a command really depends on that module being loaded.

3.28 Why modules matter so much

On a personal machine, you may be used to software simply “being installed.”

On HPC, software is often there but not active until you load the right module.

That means module management is part of basic Linux practice on the cluster, not an advanced extra topic.

3.29 Example: using a compiler

module load gcc
which gcc
gcc --version

This pattern is excellent because it confirms:

  • the module loaded
  • the executable is now visible
  • the version is what you expect

3.30 Disk usage and quotas

3.30.1 Check filesystem space

df -h

Use this when you want a broad view of available space on mounted filesystems. If a job starts failing while writing output, this command helps you check whether the problem might be lack of free space rather than a code bug.

3.30.2 Check directory sizes

du -sh *

Use this when you want to know which files or directories are consuming space in your current location. This is especially useful when a project directory becomes unexpectedly large and you need to find the main offenders quickly.

These commands answer different questions.

  • df -h shows overall filesystem space
  • du -sh * shows how much space files and directories are using in your current location

You need both perspectives.

If a job starts failing when writing output, storage problems are a real possibility.

3.31 Why storage awareness matters

Students often discover quota or storage trouble late, after:

  • copying too much data
  • writing many output files
  • saving large containers in the wrong place
  • leaving temporary data to accumulate

A good habit is to inspect storage before it becomes an emergency.

3.32 Processes

3.32.1 Show your processes

ps -u $USER

This shows the processes currently running under your username. It is useful when you are trying to confirm whether something is still running, whether you accidentally launched a task on the login node, or whether a process you expected to finish is still active.

3.32.2 Kill a process

kill <pid>

Use this when you need to stop a process by its process ID. This is most useful for accidental or lightweight shell-side processes; for scheduled jobs, you will usually use SLURM commands such as scancel instead.

3.33 Repeating a command with watch

Sometimes you do not want to run the same command over and over by hand. You want to see how something changes over time.

That is exactly what watch is for.

3.33.1 Basic use

watch squeue -u $USER

This reruns the command every few seconds and refreshes the screen so you can watch the output change.

This is especially useful when:

  • you are waiting for a job to leave the queue
  • you want to see a running job disappear when it finishes
  • you want to monitor file growth, disk usage, or another changing value

3.33.2 Why watch is useful on HPC

On an HPC system, many things change while you are working:

  • a queued job becomes running
  • a running job completes
  • a log file grows
  • disk usage increases during output generation

Instead of typing the same command repeatedly, watch lets you see those changes live.

3.33.3 Choose a shorter interval

watch -n 2 squeue -u $USER

The -n 2 option tells watch to refresh every two seconds.

That is often a good compromise: fast enough to be useful, but not so fast that the display becomes noisy.

3.33.4 Another good example

watch -n 2 'ls -lh output/'

This lets you see whether files in a directory are appearing or growing while a workflow is running.

3.33.5 A caution about watch

watch is useful because it saves effort, but it is still repeatedly running a command. Use it deliberately rather than leaving many watch sessions running without reason.

3.34 Measuring runtime with time

Another very useful command is time.

Use it when you want a quick answer to a simple question:

How long did that command actually take?

3.34.1 Basic use

time ./hello

This runs the command and reports timing information after it finishes.

For small tests, this is often the simplest way to compare one version of a command or program against another.

3.34.2 Why time matters

Even before you get to formal performance analysis, you should start building the habit of measuring instead of guessing.

For example, time can help you compare:

  • one program version versus another
  • one compiler option versus another
  • one input size versus another
  • one small serial test before moving to a larger run

3.34.3 A practical example

time grep "error" big_log.txt

That example is small, but it reinforces the larger idea: Linux can help you measure behavior directly instead of relying on intuition.

3.34.4 A note for later chapters

time is especially useful later when you begin comparing:

  • serial and parallel programs
  • optimized and non-optimized builds
  • different workflow designs

At this stage, you do not need a deep theory of performance. You just need the habit of checking real runtime when it matters.

This is especially useful if you accidentally start something on the login node that should not keep running there.

Later, SLURM job management becomes more important than plain process management, but this is still good Linux knowledge.

3.35 Working from inside R

If you use R, remember that environment information can also be inspected from inside R.

Sys.getenv("PATH")

You can also run shell commands from R:

system("echo $HOME")

This is useful because R workflows on HPC still depend on the surrounding Linux environment.

3.36 A debugging mindset for Linux

When something goes wrong, avoid random changes. Use a sequence.

3.36.1 1. Where am I?

pwd

This is the first grounding question. Before you do anything more complicated, make sure you know your current directory.

3.36.2 2. What files are here?

ls -la

This helps you verify whether the files you think are present are actually present, and whether any hidden files might matter.

3.36.3 3. Do I have the right permissions?

ls -l script.sh

If a script will not run, a file will not open, or a directory seems inaccessible, inspect permissions before changing anything else.

3.36.4 4. Is the command available?

which gcc

This checks whether the command can actually be found in the current environment. It is much better than assuming a program is installed just because you expect it to be.

3.36.5 5. Is the right environment loaded?

module list

This checks whether the software environment you intended to use is actually active. Many “mystery” failures are really just environment mismatches.

3.36.6 6. Am I out of space?

df -h
du -sh .

These two checks help you decide whether the problem is global free space or local directory growth. Storage issues can look like program failures if you are not watching for them.

That checklist is much more useful than reacting emotionally to an error message.

3.37 Mini case study: “command not found”

Suppose you run:

gcc --version

and get:

bash: gcc: command not found

A good response is:

  1. check module list
  2. load the compiler if needed: module load gcc
  3. verify with which gcc
  4. rerun the command

This is a small example, but it captures the larger lesson:

Many Linux problems are environment problems.

3.38 Mini case study: wrong directory

Suppose your job script expects input.csv, but the program cannot find it.

Before rewriting the code, check:

pwd
ls
find . -name "input.csv"

Very often, the file exists but your current directory assumption is wrong.

3.39 Common mistakes

  • forgetting chmod +x
  • working in the wrong directory
  • assuming software is available without checking modules
  • misunderstanding PATH
  • filling $HOME with large files
  • leaving important temporary data only in scratch
  • searching blindly instead of using find, head, tail, and grep
