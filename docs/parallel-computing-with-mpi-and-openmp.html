Chapter 6 Parallel Computing with MPI and OpenMP

6.1 Overview

In this chapter, we move from single-core programs to parallel computing.

You will learn:

  • the difference between serial and parallel execution
  • the difference between processes and threads
  • how MPI and OpenMP work
  • how to run parallel jobs with SLURM
  • how to think about performance and scaling

6.2 Mental Model: Why Parallel Computing?

A single core can only do one thing at a time.

Parallel computing allows you to:

  • split a problem into pieces
  • solve those pieces simultaneously
  • combine the results

6.3 Visual: Serial vs Parallel

Serial:
Task1 → Task2 → Task3 → Task4

Parallel:
Task1
Task2   → run at the same time
Task3
Task4

6.4 Two Main Models

6.4.1 MPI (Message Passing Interface)

  • multiple processes
  • can run across multiple nodes
  • processes communicate via messages

6.4.2 OpenMP

  • multiple threads
  • usually within one node
  • shared memory model

6.5 Mental Model: Processes vs Threads

MPI:
Process 1 → separate memory
Process 2 → separate memory

OpenMP:
Thread 1 → shared memory
Thread 2 → shared memory

🧠 Key Insight

MPI is for distributed computing across nodes. OpenMP is for shared memory within a node.

6.6 Loading MPI

module load openmpi

6.7 MPI Hello World (C)

Create a file:

nano hello_mpi.c

Add:

#include <mpi.h>
#include <stdio.h>

int main(int argc, char** argv) {
    MPI_Init(&argc, &argv);

    int rank, size;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);

    printf("Hello from process %d of %d\n", rank, size);

    MPI_Finalize();
    return 0;
}

6.8 Compile MPI Program

mpicc hello_mpi.c -o hello_mpi

6.9 Run Locally for Testing

mpirun -np 4 ./hello_mpi

⚠️ Important

On HPC systems, production runs should go through SLURM.

6.10 MPI Job Script

Create:

nano mpi.slurm

Add:

#!/bin/bash
#SBATCH -J mpi_test
#SBATCH -N 1
#SBATCH -n 4
#SBATCH -t 00:05:00
#SBATCH -p normal

module load openmpi

mpirun ./hello_mpi

Submit:

sbatch mpi.slurm

6.11 Expected Output

Hello from process 0 of 4
Hello from process 1 of 4
Hello from process 2 of 4
Hello from process 3 of 4

6.12 OpenMP Example

Create:

nano hello_omp.c

Add:

#include <stdio.h>
#include <omp.h>

int main() {
    #pragma omp parallel
    {
        int id = omp_get_thread_num();
        printf("Hello from thread %d\n", id);
    }
    return 0;
}

6.13 Compile OpenMP Program

gcc -fopenmp hello_omp.c -o hello_omp

6.14 Run OpenMP

export OMP_NUM_THREADS=4
./hello_omp

6.15 OpenMP SLURM Script

Create:

nano omp.slurm

Add:

#!/bin/bash
#SBATCH -J omp_test
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -c 4
#SBATCH -t 00:05:00
#SBATCH -p normal

export OMP_NUM_THREADS=4

./hello_omp

Submit:

sbatch omp.slurm

6.16 Hybrid Concept

You can combine MPI and OpenMP:

  • MPI across nodes
  • OpenMP within nodes

6.17 Mental Model

Node 1:
  Process 1 → Threads
  Process 2 → Threads

Node 2:
  Process 3 → Threads
  Process 4 → Threads

6.18 Performance Thinking

We care about how runtime changes as cores increase.

6.18.1 Strong Scaling

Same problem size, more cores.

6.18.2 Weak Scaling

Problem size increases with cores.

6.18.3 Simple Timing Example

time ./hello

💡 Insight

More cores do not always mean faster results. Communication overhead matters.

6.19 Common Mistakes

  • using MPI without loading the module
  • mismatch between -n and program expectations
  • forgetting OMP_NUM_THREADS
  • running MPI directly on the login node
  • misunderstanding cores vs tasks

6.20 Debugging Checklist

6.20.1 Check job:

squeue -u $USER

6.20.2 Check output:

cat slurm-*.out

6.20.3 Check modules:

module list

6.20.4 Check resource request:

  • nodes (-N)
  • tasks (-n)
  • cores (-c)

6.24 Key Takeaways

  • Parallel computing splits work across cores
  • MPI uses processes across nodes
  • OpenMP uses threads within a node
  • SLURM controls resource allocation
  • Performance depends on scaling and overhead
